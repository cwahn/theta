# Encode / Decode — Binary Codec for theta Remote

## Problem

`ActorRef<T>` currently implements `serde::Serialize` / `serde::Deserialize` (in `theta/src/remote/serde.rs`).
This leaks a false signal: any crate can assume `ActorRef` is a regular serializable type.
In reality, serialization only works inside a `PEER` task-local scope — calling it outside panics or produces wrong results.

Types that *contain* `ActorRef` (in a `Vec`, `Option`, struct field, etc.) inherit
the same `Serialize` bound through serde's derive.  This is incorrect: those types
are only meaningfully serializable inside theta's remote transport, not in general.

## Solution

Replace `serde::Serialize` / `serde::Deserialize` on `ActorRef` with two new traits:

```rust
// theta/src/remote/codec.rs

pub trait Encode {
    fn encode_to(&self, buf: &mut Vec<u8>) -> Result<(), EncodeError>;
}

pub trait Decode: Sized {
    fn decode_from(buf: &[u8]) -> Result<(Self, &[u8]), DecodeError>;
}
```

All 20 `postcard::to_stdvec` / `postcard::from_bytes` call-sites in the remote module
switch from serde to these traits.

## Design

### Layer 1 — Primitives (delegate to postcard)

```rust
macro_rules! impl_encode_postcard {
    ($($ty:ty),*) => { $(
        impl Encode for $ty {
            fn encode_to(&self, buf: &mut Vec<u8>) -> Result<(), EncodeError> {
                postcard::to_extend(self, buf)?; Ok(())
            }
        }
        impl Decode for $ty {
            fn decode_from(buf: &[u8]) -> Result<(Self, &[u8]), DecodeError> {
                Ok(postcard::take_from_bytes(buf)?)
            }
        }
    )* }
}
impl_encode_postcard!(
    u8, u16, u32, u64, u128,
    i8, i16, i32, i64, i128,
    f32, f64, bool, String
);
```

Postcard-delegated types keep exact binary compatibility with the existing wire format.

### Layer 2 — Containers (recursive)

```rust
impl<T: Encode> Encode for Vec<T>  { /* varint len, then T::encode_to each */ }
impl<T: Decode> Decode for Vec<T>  { /* varint len, then T::decode_from each */ }

impl<T: Encode> Encode for Option<T>  { /* 0x00 | 0x01 + inner */ }
impl<T: Decode> Decode for Option<T>

impl<T: Encode> Encode for Box<T>
impl<T: Decode> Decode for Box<T>

impl<K: Encode, V: Encode> Encode for HashMap<K, V>
impl<K: Decode + Eq + Hash, V: Decode> Decode for HashMap<K, V>

impl Encode for ()  { /* no-op */ }
impl Decode for ()

// Tuples: (A, B), (A, B, C), etc.
impl<A: Encode, B: Encode> Encode for (A, B)
impl<A: Decode, B: Decode> Decode for (A, B)
```

Length prefixes use postcard's varint encoding for wire compatibility.

### Layer 3 — ActorRef (PEER context)

```rust
impl<A: Actor> Encode for ActorRef<A> {
    fn encode_to(&self, buf: &mut Vec<u8>) -> Result<(), EncodeError> {
        ActorRefDto::from(self).encode_to(buf)  // uses PEER.with(...)
    }
}

impl<A: Actor> Decode for ActorRef<A> {
    fn decode_from(buf: &[u8]) -> Result<(Self, &[u8]), DecodeError> {
        let (dto, rest) = ActorRefDto::decode_from(buf)?;
        let actor_ref = dto.try_into()
            .map_err(|e| DecodeError::ActorRef(format!("{e}")))?;
        Ok((actor_ref, rest))
    }
}
```

`ActorRefDto` itself uses postcard-delegated Encode/Decode (it's a simple enum of `ActorId` + optional `PublicKey`).

### Layer 4 — User types (`#[data]` derive macro)

```rust
#[data]
pub struct ChatMessage {
    pub author: String,
    pub text: String,
}
```

Generates:

```rust
impl Encode for ChatMessage {
    fn encode_to(&self, buf: &mut Vec<u8>) -> Result<(), EncodeError> {
        self.author.encode_to(buf)?;
        self.text.encode_to(buf)?;
        Ok(())
    }
}
impl Decode for ChatMessage {
    fn decode_from(buf: &[u8]) -> Result<(Self, &[u8]), DecodeError> {
        let (author, buf) = String::decode_from(buf)?;
        let (text, buf) = String::decode_from(buf)?;
        Ok((Self { author, text }, buf))
    }
}
```

For enums: variant index encoded as varint, then variant fields.

```rust
#[data]
pub enum MyEnum {
    A,
    B(String),
    C { x: u32, y: u32 },
}
// Variant indices: A=0, B=1, C=2
// Encoding: [varint index] [fields...]
```

### Layer 5 — Internal protocol types

Theta's own protocol types (`InitFrame`, `ContinuationDto`, `ControlFrame`,
`ActorRefDto`, `ForwardInfo`, `Status`, `Escalation`, `MonitorError`)
manually implement `Encode`/`Decode` or use a private derive.

`Update<A>` and `MsgPackDto<A>` switch from `derive(Serialize)` to `Encode`/`Decode`.
`Msg` enum (generated by `#[actor]`) also gets `Encode`/`Decode` from the macro.

`M::Return` (message return types) must implement `Encode`/`Decode`.

## Error Types

```rust
pub enum EncodeError {
    Postcard(postcard::Error),
}

pub enum DecodeError {
    Postcard(postcard::Error),
    ActorRef(String),
    UnexpectedEnd,
    InvalidVariant(u32),
}
```

## Migration

### Removed
- `impl Serialize for ActorRef<A>` (remote/serde.rs:101)
- `impl Deserialize for ActorRef<A>` (remote/serde.rs:110)
- `#[cfg_attr(feature = "remote", derive(Serialize, Deserialize))]` on `Update<A>`, `Status`, `Escalation`
- `#[derive(Serialize, Deserialize)]` on `ActorRefDto`, `ContinuationDto`, `InitFrame`, `ControlFrame`, `ForwardInfo`, `MonitorError`
- All `postcard::to_stdvec` / `postcard::from_bytes` call-sites (20 total, listed in session notes)

### Added
- `theta/src/remote/codec.rs` — trait definitions, primitive/container impls, error types
- `impl Encode/Decode for ActorRef<A>` — in codec.rs (replaces serde impls)
- `impl Encode/Decode` for all protocol types listed above
- `#[data]` proc macro in theta-macros — derives `Encode`/`Decode` for user structs/enums
- `encode_to_vec(t: &impl Encode) -> Result<Vec<u8>, EncodeError>` convenience fn
- `decode_from_bytes<T: Decode>(buf: &[u8]) -> Result<T, DecodeError>` convenience fn

### Call-site migration pattern

```rust
// Before:
postcard::to_stdvec(&(msg, k_dto))
// After:
encode_to_vec(&(msg, k_dto))

// Before:
postcard::from_bytes::<MsgPackDto<A>>(&buf)
// After:
decode_from_bytes::<MsgPackDto<A>>(&buf)

// Before:
PEER.sync_scope(peer, || postcard::to_stdvec(&ret))
// After:
PEER.sync_scope(peer, || encode_to_vec(&ret))
```

### User-facing migration

```rust
// Before: messages and views derive Serialize/Deserialize
#[derive(Serialize, Deserialize)]
pub struct SendMessage { pub text: String }

// After: use #[data] instead (or additionally, if serde is still wanted for other purposes)
#[data]
#[derive(Serialize, Deserialize)]  // optional, for non-theta uses
pub struct SendMessage { pub text: String }
```

Types that contain `ActorRef` can no longer derive `Serialize`.
They must use `#[data]` for Encode/Decode.

## Decisions

| ID | Decision |
|----|----------|
| D1 | Trait names: `Encode` / `Decode` |
| D2 | `Encode`/`Decode` is for NETWORK boundary only (`theta::remote`). Not for JS interop. |
| D3 | JS/TS interop is ORTHOGONAL to Encode/Decode. Do not mix them. |
| D4 | `ActorRef` is NOT serde-serializable. Any type containing `ActorRef` is also not serde-serializable — that is the point and intention. |
| D5 | ALL types in theta messages must impl `Encode`. If not → compile error. |
| D6 | Structural derivation: derive walks fields, every inner type must impl `Encode`. |
| D7 | No special annotation needed for inner types — just impl `Encode` (derived or manual). |
| D8 | Types WITHOUT `ActorRef` can still `derive(Serialize)`. Types WITH `ActorRef` should not be serializable but `Encode`/`Decode`-able. |
| D9 | Remote feature will use `Encode`/`Decode` instead of serde. |
| D10 | Traits live in `theta::remote` module. |
| D11 | Scope: full replace at remote boundary, no matter how big. |
| D12 | `#[data("uuid")]` is a thin marker with type tag (like `#[actor("uuid")]`). Not heavy codegen. |
| D13 | Users implement familiar serde; proc macro derives Encode/Decode from the structure. |
| D14 | Orphan rule is irrelevant — theta defines `Encode`, can impl for any type. |
| D15 | For JS boundary: `ActorRef` is generic opaque type that needs wrapping — solved separately from Encode/Decode. |
| D16 | Keep discussion essential and optimal, step by step. |
| D17 | Binary format: use current postcard-based format as-is. Postcard format is stable. |
| D18 | Design B chosen: thin wrapper over postcard, not abstract Serializer mirror. |
| D19 | The point: don't signal to other crates that `ActorRef` is `Serialize`/`Deserialize` — it's not. |

## Non-goals

- JS/TS boundary encoding — orthogonal (D3), solved separately
- Changing the binary wire format — postcard-compatible byte layout preserved
- Removing serde from types that don't contain ActorRef — those can keep `derive(Serialize)` freely
